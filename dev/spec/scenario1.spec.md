## 2. 场景摘要

### 2.1 原始问题描述

旅游 APP 酒店预订场景中，用户选择指定酒店并进入酒店套餐选择界面时，页面当前会默认选中第一个套餐；目标是取消默认选项，让用户自行选择。

### 2.2 对应特定场景的摘要

当前分支已经具备酒店搜索结果页、酒店详情页、套餐列表和下单页。问题并非缺失页面，而是详情页状态初始化错误：进入酒店详情时自动将 `selectedPackageId` 设为首个套餐，同时套餐读取逻辑在未命中时也会回退到首个套餐，导致“未选择”状态无法成立。

### 2.3 预期行为

用户从酒店结果页进入某酒店详情后：

- 套餐列表初始为“未选择”状态
- 不出现默认高亮或默认选中的首个套餐
- 用户点击某个套餐后，才进入下单页
- 下单、支付、订单生成仍使用用户显式选择的套餐

## 4. 重建后的产品流程

### 4.1 前置条件

- 用户已进入酒店查询结果页
- 当前服务类型为 `hotel`
- 结果列表存在可点击的酒店商品

### 4.2 页面序列

首页/搜索页 -> 酒店结果页 -> 酒店详情页（套餐选择） -> 下单页 -> 支付页 -> 订单页

### 4.3 用户动作与系统响应

1. 用户在结果页点击指定酒店
2. 系统进入酒店详情页，但不预设套餐
3. 页面展示套餐列表与“请选择套餐后继续预订”的提示
4. 用户点击某一套餐的操作按钮
5. 系统记录该套餐 ID，并进入下单页
6. 后续金额、订单摘要、支付与订单写入均基于该显式选择

### 4.4 重点关注的触发点

- 进入酒店详情页时的 `selectedPackageId` 初始化
- 套餐读取逻辑在无选择时不得兜底到首个套餐
- 下单页、支付页仅在已选择套餐时展示

## 5. UI / 页面设计规格

### 5.1 涉及页面

- `results` 结果页
- `detail` 酒店详情页
- `booking` 下单页
- `payment` 支付页

### 5.2 页面级布局与分区

- 结果页保持现有商品卡片列表
- 详情页保持头图、卖点、套餐列表三段式布局
- 套餐列表增加“当前未默认选择套餐，请手动选择”的引导文案
- 已显式选中的套餐卡片可继续保留高亮态

### 5.3 关键 UI 组件与状态

- 套餐卡片状态：
  - `unselected`：默认进入详情页时所有套餐均为此状态
  - `selected`：用户主动点击后命中的套餐
- 预订按钮状态：
  - 未选中时展示“选择此套餐”
  - 已选中时展示“已选择”

### 5.4 导航与路由规则

- 保持当前 `Index.ets` 内部 `currentScreen` 导航模式
- `openProduct()` 进入 `detail`
- `startBooking()` 仅在用户点击某套餐时进入 `booking`

### 5.5 状态与数据要求

- `selectedPackageId` 必须允许空值表达“未选择”
- 套餐查询函数在空值或未命中时返回 `undefined`
- 详情页依赖酒店商品 mock 数据
- 所有关联 mock 数据拆分为独立 JSON 文件并统一存放于 `dev/mock-data/travel-001/`

## 8. 测试策略

### 8.1 功能测试用例

- 进入酒店详情页时，不应有任意套餐默认高亮
